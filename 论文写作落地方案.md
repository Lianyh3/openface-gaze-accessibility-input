## 1. 目标与定位

这篇论文不是单纯介绍“我做了一个程序”，而是要完成一篇符合本科毕业设计要求的**工程型论文**。全文必须同时体现以下三点：

1. **分析**：把“眼控输入”问题抽象为可求解的计算机系统问题。
2. **设计**：给出系统架构、关键模块、算法流程、接口设计。
3. **验证**：用实验和指标证明系统可运行、可比较、可改进。

结合当前项目，论文的核心主线建议固定为：

- 以 OpenFace 为 baseline 信号源；
- 通过标定、平滑、命中测试、dwell 触发、候选词与运行时迁移，构建一套可运行的无障碍眼控输入系统；
- 通过实时性、输入效率和一致性实验，证明方案具有工程价值。
- 写作重点放在“系统如何分析、如何设计、为什么这样设计、设计带来了什么效果”，而不是强调具体使用了哪一种编程语言。

## 2. 题目建议（建议尽快冻结）

模板要求题目原则上不超过 20 字，因此建议从下面几个版本中选一个作为最终题目：

1. **基于OpenFace的实时眼控输入系统设计**
2. **面向无障碍交互的实时眼控输入系统研究**
3. **实时眼控输入系统的设计与实现**

如果希望突出“优化”，可使用：

4. **基于OpenFace的眼控输入优化研究**

建议优先选择第 1 个，因为：

- 与现有代码和实验最匹配；
- 既包含 OpenFace，又包含系统设计；
- 容易围绕“设计—实现—验证”展开；
- 比“方法研究”更适合本科工程型毕业设计。

## 3. 写作总策略

### 3.1 写作顺序建议

不要从第 1 章顺写。推荐顺序如下：

1. 先写第 4 章“系统实现”；
2. 再写第 5 章“实验与结果分析”；
3. 再写第 3 章“系统分析与总体设计”；
4. 然后写第 2 章“相关技术与理论基础”；
5. 最后写第 1 章“绪论”、摘要、结论、致谢。

原因：

- 第 4、5 章最贴近现有代码和实验结果，最容易落笔；
- 第 1 章和摘要必须在“全文已成型”后才能写得准确；
- 结论要基于实验结果，不能先写。

### 3.2 全文逻辑主线

建议全文围绕下面一句话组织：

> 本文面向无障碍交互场景，构建了一套基于 OpenFace 的实时眼控输入系统，并围绕视线信号稳定化、注视触发输入、候选词辅助与运行时架构优化进行了设计与验证。

只要每一章都服务这句话，论文就不会散。

## 4.1 写作总原则（本次调整）

从论文表达上，后续统一遵循下面三条原则：

1. **先讲设计目标，再讲实现路径**：先说明模块要解决什么问题，再说明采用什么机制。
2. **先讲系统思路，再讲工程落地**：重点描述模块划分、数据流、状态机、触发逻辑、实验验证，不强调语言本身。
3. **语言只是实现载体，不是论文主角**：除非在“运行时架构优化”一节中讨论性能与分层，否则不主动突出具体编程语言。

因此，后续每一节写作时，都尽量围绕以下四个问题展开：

- 为什么要设计这个模块？
- 这个模块的输入、输出和内部机制是什么？
- 为什么采用这种设计而不是其他设计？
- 这种设计最终带来了什么实验效果或工程收益？

### 5.2 中文摘要

模板要求：摘要一般 400 字以内，不分段，不出现公式、图表、引文标注。

#### 中文摘要写法公式

建议严格按下面四步写：

1. **研究背景与意义**：眼控输入适用于哪些场景，为什么重要；
2. **研究问题**：OpenFace 原始 gaze 信号在实时输入场景中存在什么问题；
3. **研究工作**：本文做了哪些模块设计与实现；
4. **实验结果**：系统达到了什么效果。

#### 你的摘要应该包含的具体内容

可以围绕以下信息组织：

- 研究对象：基于摄像头的人机交互眼控输入系统；
- 背景：传统键鼠对肢体障碍用户不友好，眼控输入具有现实意义；
- 问题：原始 gaze 信号抖动大、命中不稳定、误触发多、实时输入效率有限；
- 方法：构建 OpenFace 实时采集链路，引入线性归一化 / 标定、时序平滑、hit-test、dwell、候选词事件流，以及 C++ 运行时迁移机制；
- 结果：实现了可运行的实时眼控输入闭环，并完成离线样例、固定文本任务、Python/C++ 后端一致性等验证。

#### 中文摘要落笔模板

可以按下面结构改写成正式摘要：

> 面向无障碍交互场景，针对基于摄像头的眼控输入系统在实时使用中存在的注视点抖动、命中不稳定和误触发等问题，本文设计并实现了一套基于 OpenFace 的实时眼控输入系统。首先，构建了 OpenFace 视线数据采集与解析链路；其次，围绕 gaze 信号处理设计了归一化映射、个体标定、时序平滑、注视区域命中测试和 dwell 触发机制；然后，结合虚拟键盘事件流与候选词机制，实现了文本输入闭环；最后，采用 Python 与 C++ 混合架构完成关键运行时模块迁移，并通过一致性与性能对比验证系统实现效果。实验结果表明，该系统能够稳定完成固定文本输入任务，在输入准确性、实时性和系统可运行性方面满足毕业设计要求，为面向无障碍交互的低成本眼控输入提供了一种可行方案。

### 5.6 第1章 绪论

这是老师最先看的一章，必须写得像“论文”，而不是项目说明书。

#### 建议结构

- 1.1 研究背景与意义
- 1.2 国内外研究现状
- 1.3 研究内容与技术路线
- 1.4 论文结构安排

#### 1.1 研究背景与意义写什么

回答三个问题：

1. 为什么眼控输入值得研究？
2. 为什么无障碍场景是重要应用场景？
3. 为什么基于普通摄像头 + OpenFace 的方案有价值？

可以写的点：

- 无障碍交互需求增长；
- 专用眼动仪成本高，摄像头方案成本低；
- OpenFace 提供现成的人脸关键点和 gaze 基线，适合作为工程起点；
- 但直接用于输入仍有抖动、误触发、稳定性不足等问题。

#### 1.2 国内外研究现状写什么

建议拆成三类文献综述：

1. 视线估计与 gaze tracking 研究；
2. 眼控输入与虚拟键盘研究；
3. 实时系统、校准、平滑与误触发抑制研究。

写法要求：

- 按时间脉络写；
- 不能只堆文献摘要；
- 每段最后要落到“现有研究还缺什么”；
- 最终引出你的工作定位：不是重新发明 gaze estimation，而是做面向工程落地的实时眼控输入闭环优化。

#### 1.3 研究内容与技术路线写什么

这部分要明确说明你做了哪些工作。

建议写成 4 个点：

1. 构建 OpenFace gaze 数据采集与解析链路；
2. 设计 gaze 命中测试、dwell 触发和虚拟键盘事件流；
3. 引入标定、平滑与候选词辅助机制，提高输入稳定性和可用性；
4. 将关键运行时迁移为 C++ 核心，构建 Python/C++ 混合架构并完成一致性验证。

同时放一张**技术路线图**。

#### 1.4 论文结构安排写什么

按“第1章……；第2章……；第3章……”逐章概述，每章 1–2 句即可。

### 5.7 第2章 相关技术与理论基础

#### 每节怎么写

**2.1 OpenFace 视线估计基础**

- OpenFace 能输出什么字段；
- 关键点、头姿态、gaze_angle 的意义；

这一章最贴近你现有系统，但写作时不要按源码文件逐个介绍，而要按“模块目标—设计思路—关键机制—实现效果”来组织。

#### 4.5 运行时架构优化与后端切换设计

这是你论文里比较有亮点的一节，但重点不应放在“用了什么语言”，而应放在“为什么这样划分运行时职责”。

建议写：

- 实时链路如何支持不同后端切换；

这一节建议配 2 张图：

1. 运行时分层架构图；
2. 后端切换与回退流程图。

### 5.10 第5章 实验设计与结果分析

这是最重要的一章，决定论文是否“像一篇完整论文”。

#### 建议结构

- 5.1 实验环境与实验条件
- 5.2 OpenFace baseline 链路验证
- 5.3 输入功能与可用性验证
- 5.4 标定与平滑效果分析
- 5.5 Python 与 C++ 后端一致性 / 性能对比
- 5.6 消融实验与结果讨论
- 5.7 本章小结

#### 5.1 实验环境与实验条件

应列成表格：

- 操作系统
- CPU / 内存
- 摄像头
- OpenFace 版本
- Python / g++ 版本
- 采样参数
- dwell 阈值
- 是否启用标定 / 平滑

#### 5.2 OpenFace baseline 链路验证

写什么：

- 是否能稳定输出 CSV；
- gaze/pose 字段是否完整；
- 帧数、时长、成功率；
- 用样例图与摄像头运行证明链路成立。

#### 5.3 输入功能与可用性验证

写什么：

- 固定文本输入任务；
- 输出指标：CER、CPM、WPM、exact match rate；
- 可以选 3–5 句固定测试句做多轮输入；
- 展示系统确实能完成输入闭环。

#### 5.4 标定与平滑效果分析

写什么：

- 不标定 vs 标定；
- 不平滑 vs EMA vs One Euro；
- 比较误触发、命中稳定性、输入效率；
- 说明不同参数下的变化趋势。

#### 5.5 不同运行时后端的一致性与性能对比

写什么：

- 比较不同运行时后端输出事件序列是否一致；
- 比较不同后端在处理耗时上的差异；
- 强调本节的核心不是语言优劣，而是验证“分层架构是否成立、切换是否可行、结果是否一致”。

#### 5.6 消融实验与结果讨论

建议至少做 4 组：

1. 去掉标定；
2. 去掉平滑；
3. 去掉候选词辅助；
4. 切换到另一种运行时后端。

讨论重点：

- 哪个模块对稳定性最有帮助；
- 哪个模块对输入效率提升最明显；
- 当前系统瓶颈在哪里。

### 5.11 结论

结论不能只是“我完成了这个系统”。

建议结构：

1. 总结本文围绕什么问题开展研究；
2. 分点概括完成了哪些工作；
3. 概括实验结果说明系统达到了什么效果；
4. 说明仍然存在的不足；
