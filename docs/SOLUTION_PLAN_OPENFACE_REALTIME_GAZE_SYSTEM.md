## 2. 总体目标与可交付成果

### 2.1 总体目标

1. 通过对 OpenFace 视线输出进行平滑、标定与触发策略优化，可显著提升眼控输入稳定性；

## 3. 三层架构设计

## 3.1 算法优化层

### 3.1.1 目标
将 OpenFace 原始输出转换为可用于交互的稳定 gaze 点与触发事件。

### 3.1.2 核心模块
- `openface_stream`：读取实时数据源（已确定采用 **B方案：OpenFace CSV 持续写入 + 轮询读取**）
- `calibrator`：九点标定与屏幕映射
- `smoother`：平滑（EMA/OneEuro 二选一或可切换）
- `quality_gate`：低置信度过滤与异常点剔除
- `dwell_state_machine`：停留触发策略（enter/stay/fire/cancel）

### 3.1.3 关键输出
- 屏幕坐标 gaze 点（x, y）
- 点质量分数（confidence/validity）
- 触发状态事件（dwell 相关）

---

## 3.2 系统实现层

### 3.2.1 目标
完成实时眼控虚拟键盘闭环：
**视线输入 -> 命中键位 -> 触发输入 -> 文本输出**。

### 3.2.2 核心模块
- `keyboard_layout`：键位模型与布局参数
- `hit_tester`：gaze 点命中检测
- `input_controller`：文本编辑状态机（字符、空格、退格、确认）
- `runtime`：实时主循环与模块编排
- `logger`：会话日志与实验记录

### 3.2.3 工程要求
- 低耦合：算法层与 UI 层解耦
- 可替换：键盘布局、触发参数可配置
- 可回放：每次会话可重放分析

---

## 3.3 智能交互增强层

### 3.3.1 目标
在不改变基础输入机制的前提下，增加“候选词补全”能力。

### 3.3.2 核心模块
- `prompt_builder`：构造候选词补全提示词
- `codex_client`：统一封装 `gpt-5.3-codex API` 调用
- `candidate_engine`：候选去重、排序、缓存、超时降级

### 3.3.3 增强策略
- 用户输入前缀达到阈值后触发候选请求；
- 返回 Top-K 候选（建议 3~5）；
- gaze 可直接选择候选词插入；
- API 失败时自动降级为纯键盘模式，确保系统不中断。

## 9. OpenFace 接入方案（已定稿）

### 9.1 主方案：B 方案（CSV 持续写入 + 轮询读取）
本项目采用 OpenFace 输出 CSV 文件持续写入方式，系统端以固定周期轮询增量读取新行，形成伪实时输入流。

### 9.2 选择理由
1. **工程风险可控**：实现复杂度低于实时流式 SDK/管道方案；
2. **兼容性高**：与 OpenFace 常见使用流程一致；
3. **可调优**：可通过轮询周期、缓冲策略、去重机制控制延迟；
4. **可复现**：同一 CSV 可用于离线重放，有利于实验复验。

### 9.3 实施要点
- 轮询周期建议：`30~50ms`（后续按实测优化）；
- 增量读取：基于文件偏移量（offset）读取新增行；
- 数据去重：按时间戳/帧号去重，防止重复消费；
- 容错处理：当写入中断或行不完整时跳过并等待下一周期；
- 同步策略：算法层统一使用单调时间戳，避免跨线程时序错乱。

### 9.4 备选与兜底
- **C 方案（录制回放）**：用于实验复现与调参，不作为主交互链路；
- **A 方案（实时推流）**：作为未来优化方向，不纳入当前毕业设计主线。

## 10. 风险与应对

1. **实时延迟过高**
   - 应对：优化轮询周期、异步请求、API 超时降级、本地缓存
2. **CSV 读写竞争导致丢帧/脏读**
   - 应对：基于 offset 增量读取 + 行完整性校验 + 失败重试
3. **误触较多**
   - 应对：调大 dwell、加入目标吸附或 hit 扩展区
4. **标定漂移**
   - 应对：支持快速重标定与会话中校正
5. **实验样本不足**
   - 应对：同一被试多轮任务 + 详细日志与案例分析

## 11. 验收标准与通过阈值（建议写入答辩材料）

为避免“功能完成但不可判定”的问题，建议定义如下验收门槛：

### 11.1 系统可运行性（必须通过）
1. 可连续运行 `30 min` 以上无崩溃；
2. 实时链路可持续输出有效 gaze 点；
3. 虚拟键盘可完成字符输入、空格、退格、确认；
4. API 不可用时可自动降级为无 AI 模式。

### 11.2 实时性（建议阈值）
1. gaze 处理链路平均延迟 < `200 ms`；
2. dwell 触发响应稳定，无明显卡顿；
3. AI 候选返回 `p95 < 1000 ms`（超时走降级）。

### 11.3 输入效果（建议阈值）
1. 固定任务句输入完成率 ≥ `90%`；
2. 相比 Baseline，AI 增强组在 CPM 或任务时长上有可观改善；
3. 误触率可控（由实验数据给出最终阈值）。

## 13. 提示词与候选策略规范（AI 模块）

### 13.1 提示词固定结构
- `system_prompt`：定义角色、输出限制、语言风格；
- `task_prompt`：给出当前前缀、上下文、候选数量要求；
- `output_constraint`：严格要求结构化输出（例如 JSON 数组）。

### 13.2 候选生成约束
1. 候选数量固定为 Top-K（建议 3~5）；
2. 候选去重并过滤过长短语；
3. 若前缀过短（如 <2 字符）可不触发请求；
4. 同一前缀短时间内优先读缓存，减少 API 调用；
5. API 超时/失败时返回空候选并不中断主输入流程。

### 13.3 中文场景建议
由于论文与任务偏中文，建议默认以中文词语/短语补全为主，英文作为可选模式。
