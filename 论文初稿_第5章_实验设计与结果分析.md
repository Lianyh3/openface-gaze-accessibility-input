# 第5章 实验设计与结果分析

本章围绕“方法是否在真实交互中有效”开展验证。与第3章方法描述和第4章系统实现不同，本章核心任务是建立可复核的实验口径，并用过程数据解释结果来源。实验数据来自本课题的阶段性日志与最新批量实验报告，覆盖基线链路、闭环输入、一致性与失败诊断等关键环节。

## 5.1 实验目标、条件与评价口径

本章对应四个研究问题：`RQ1`（基础链路是否稳定可用）、`RQ2`（优化链路接入后能否完成固定文本输入）、`RQ3`（不同运行实现是否保持一致语义）、`RQ4`（失败主要发生在哪一层）。这四个问题共同覆盖“可观测、可完成、可复现、可解释”四个维度，与本文研究目标保持一致。

实验环境为 Ubuntu 系统，OpenFace 版本为 `2.2.0`，输入设备为普通摄像头。系统默认停留阈值为 `600 ms`，并使用第3章定义的归一化、标定和平滑接口。为保证结论可追溯，实验数据按用途分为六类：基础链路数据、闭环任务数据、一致性校验数据、失败会话数据、平滑策略消融数据和最新批量实验数据。

本章采用“链路指标 + 任务指标 + 一致性指标”三层评价。链路指标用于描述 OpenFace 基础观测质量：

`success_rate = success_frames / total_frames`  （5.1）

并结合 `mean_confidence`、`estimated_fps`、`p95_frame_interval_ms` 判断实时稳定性。任务指标用于评估文本输入效果：

`CER = edit_distance(output, target) / max(1, len(target))`  （5.2）

`CPM = N_char / (T/60),  WPM = CPM / 5`  （5.3）

其中 `output` 以提交区文本为主口径，组合区文本作为诊断口径，用于区分“识别失败”和“交互未闭环”。一致性指标用于验证跨后端可替换性：

`mismatch_count = count(event_or_value_not_equal)`  （5.4）

当 `mismatch_count = 0` 时，说明在同一输入下两后端具备等价行为语义，可支持后续架构演进。

为避免伪差异，本章比较统一采用三条口径：任务文本一致（固定句）、事件统计一致（`key_input / candidate_pick`）、时长基准一致（`event.metrics.emitted_at_ms`）。统一时基尤其关键，因为部分早期日志仅含 `timestamp_utc`，会导致 `CPM/WPM` 异常放大。

## 5.2 基础链路与闭环任务结果

基础链路方面，静态样例 `sample1.csv` 共 `1` 帧，成功帧 `1`，`success_rate = 1.0`，`mean_confidence = 0.98`，说明在理想条件下 OpenFace 可稳定输出 gaze 与姿态字段。

在 `2026年4月1日` 完成的最新批量实验中，baseline 部分共包含 `5` 次摄像头采集，每次约 `25 s`。聚合结果为：`mean_success_rate = 1.0`，`mean_confidence = 0.9316`，`mean_estimated_fps = 28.83`，`mean_duration_seconds = 24.46`；单次 `success_rate` 范围为 `0.9788~1.0`。同时 `valid_gaze_rows` 与 `valid_pose_rows` 持续完整，说明当前链路瓶颈不在字段缺失，而在采集条件波动控制。

闭环任务方面，早期单次样例能够完成固定目标句输入，结果为 `CER = 0`、`exact_match_rate = 1.0`、`CPM = 70.59`、`WPM = 14.12`、`mean_dwell_elapsed_ms = 680`。事件统计为 `key_input = 4`、`candidate_pick = 1`，最终提交文本为“我今天想去”，表明“坐标序列 -> 事件流 -> 提交文本”的闭环已形成。

在统一实验界面、统一阈值和统一评价口径下，最新一轮批量端到端实验包含两类固定文本任务（`T01/T02`）共 `6` 轮，聚合结果为：`run_count = 6`，`exact_match_rate = 1.0`，`mean_CER = 0.0`，`mean_CPM = 82.0`，`mean_WPM(5char) = 16.4`。按实现分组，A/B 两种运行实现均保持 `exact_match_rate = 1.0` 与 `mean_CER = 0.0`；按任务分组，`T01` 的 `mean_CPM = 80.0`、`T02` 的 `mean_CPM = 84.0`，总体差异可解释为任务路径长度差异。

## 5.3 一致性验证与消融分析

一致性校验显示，两种实现输出事件序列均为 `5` 条，`mismatch_count = 0`，且 `event_type/target_id/dwell_elapsed_ms` 一致，说明命中与停留核心逻辑在跨实现条件下保持语义等价。

在数值层面，`m2_alignment_check_report.json` 中标定与平滑比较同样为 `mismatch_count = 0`。标定拟合指标为 `mae_x = 0.002606`、`mae_y = 0.002418`、`rmse = 0.002783`、`max_abs_error = 0.004844`；EMA 与 One-Euro 前序轨迹点在两后端下逐点一致，未观察到跨语言附加漂移。运行时间对比（`repeat = 5`）显示实现A平均耗时 `152.49 ms`、实现B平均耗时 `159.68 ms`，均值差 `-7.19 ms`，在当前短序列样例上尚无显著性能优势。

为评估平滑模块的独立贡献，本文在同一输入序列、同一停留阈值（`600 ms`）下进行三组消融：`none`、`ema(alpha=0.4)`、`one_euro(min_cutoff=1.0, beta=0.01, d_cutoff=1.0)`。核心结果如下：

`none`：`event_count = 5`，`candidate_pick = 1`，`CER = 0`，`CPM = 70.59`，成功提交“我今天想去”；  
`ema`：`event_count = 3`，`candidate_pick = 0`，`CER = 1.0`，未提交，停留均值 `716.67 ms`；  
`one_euro`：`event_count = 4`，`candidate_pick = 0`，`CER = 1.0`，未提交，停留均值 `687.5 ms`。

结果表明，平滑并非在所有参数下都提升任务指标。当前参数组合下，`ema` 与 `one_euro` 引入的滞后使候选区命中不足，最终未形成提交闭环。这与第3章结论一致：平滑强度必须与触发阈值联合调参，不能孤立设置。

## 5.4 失败案例与有效性讨论

早期一次失败会话记录显示：事件总数 `4`，仅有 `key_input`，无 `candidate_pick` 或提交动作；最终状态为 `committed_text = ""`、`composing_buffer = "我今天想去"`。按提交区口径计算，`CER = 1.0`、`CPM = 0`。该案例说明失败并非来自字符识别错误，而是发生在“提交闭环未完成”阶段，也进一步验证了双口径评估的必要性。

结合成功与失败会话，本章将问题归因为三层：感知层（观测质量波动导致边界跳变风险上升）、触发层（阈值与抑制逻辑影响事件稳定性）、提交层（候选确认与提交动作决定任务是否闭环）。当前失败样例主要暴露提交层薄弱，而非感知层完全失效，因此后续应优先加强“未提交提醒”“提交确认反馈”等交互机制。

本章结论仍存在边界：目前虽已完成 `5` 次 baseline 与 `6` 次端到端批量实验，但任务规模和被试规模仍偏小，场景主要集中于短句输入，尚不足以支持广泛人群下的统计最优结论。后续仍需补充多被试、多场景、多轮次实验以增强稳健性。

## 5.5 本章小结

本章基于统一口径的真实运行报告，完成了从基础链路到端到端任务的分层验证。结果表明：OpenFace 基础链路在当前条件下具备较高稳定性；端到端批量实验在 `6` 轮任务上达到 `exact_match_rate = 1.0` 且 `mean_CER = 0.0`；两种运行实现在事件语义和关键数值上保持一致；平滑消融进一步证明参数配置会直接影响提交闭环成功率；失败案例主要集中在提交层而非字符识别层。上述结论为第6章总结与后续优化方向提供了直接依据。
