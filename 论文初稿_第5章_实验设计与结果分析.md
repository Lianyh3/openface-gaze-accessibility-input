# 第5章 实验设计与结果分析

## 5.1 实验目标与评价方法

本章实验围绕两个研究问题展开：其一，第3章提出的视线估计优化链路在实时条件下是否稳定；其二，第4章构建的眼控输入系统能否基于该链路完成可量化的文本输入任务。为回答上述问题，本文将实验划分为基础链路验证、闭环输入验证、一致性验证与案例分析四个部分。

评价指标采用“过程指标+结果指标”的组合方式。过程指标包括成功率、置信度、估计帧率和帧间隔统计，用于描述基础感知链路质量；结果指标包括字符错误率（CER）、输入速度（CPM、WPM）和触发时长统计，用于衡量任务完成效果。其中：

`CER = edit_distance(output, target) / max(1, len(target))`  （5.1）

`CPM = N_char / (T/60),  WPM = CPM / 5`  （5.2）

由式（5.1）和式（5.2）可见，文本质量指标与效率指标分别对应“正确性”与“可用性”两个维度。上述定义保证了方法效果可以被不同会话与不同后端口径统一比较。

## 5.2 实验条件与数据组织

实验在 Ubuntu 环境下完成，视线数据由普通摄像头与 OpenFace 2.2.0 提供。系统停留阈值设置为 `600 ms`，并按第3章方法执行归一化、标定和平滑流程。为降低人工记录误差，实验过程统一采用结构化报告和会话日志进行保存。

本章使用的数据可分为三类。第一类是基础链路数据，包括静态样例与摄像头实时采集结果，用于验证 OpenFace 输出可用性。第二类是闭环输入数据，包括固定短文本任务会话，用于计算 CER 与输入速度。第三类是一致性与对比数据，包括 Python 与 C++ 后端对齐报告和重复运行时间统计，用于验证不同后端在输出语义与时间开销上的差异。

## 5.3 基础链路实验结果

静态样例实验中，系统共处理 `1` 帧，成功帧 `1`，成功率 `1.0`，平均置信度 `0.98`。该结果说明在理想静态条件下，OpenFace 可稳定输出完整 gaze 与姿态字段。

摄像头实时实验共记录 `383` 帧，其中成功帧 `254`，成功率 `0.6632`，平均置信度 `0.6185`，平均估计帧率 `19.65 FPS`，平均持续时长 `19.445 s`，`p95` 帧间隔为 `139.95 ms`。与静态样例相比，实时链路受头姿与环境变化影响更加明显，说明原始观测信号在真实交互前需要进一步优化处理，这一现象与第3章问题建模一致。

## 5.4 闭环输入实验结果

在固定任务“我今天想去”的闭环实验中，单次会话包含 `15` 个输入点，全部实现有效映射，触发事件总数为 `5`，其中字符输入 `4` 次、候选选择 `1` 次，分发错误数为 `0`。最终文本状态为成功提交“我今天想去”，表明从连续坐标到离散事件再到文本输出的流程能够完整闭环。

在同一任务上，Python 与 C++ 两种后端均得到一致的任务指标：`CER = 0`，`exact match = 1.0`，`CPM = 70.59`，`WPM(5char) = 14.12`，平均停留触发时长 `680 ms`。该结果说明，在当前参数设置下，系统能够在准确性与触发稳定性之间达到可用平衡。

候选行为统计显示，会话中发生 `1` 次候选选择，候选曝光总次数为 `32`，覆盖 `9` 个候选词。该结果表明候选机制在短文本任务中能够降低逐字输入负担，并对最终提交效率形成正向支撑。

进一步从事件路径分析可见，任务完成依赖“4次字符输入 + 1次候选确认”的组合。若仅保留字符输入而缺少候选确认或提交动作，输出状态会停留在组合区，导致最终指标下降。由此可知，闭环性能不仅受坐标质量影响，也受事件组织策略影响。第4章中“触发机制—文本状态管理—候选确认”三者耦合关系，在本实验中得到了验证。

此外，停留时长均值 `680 ms` 与默认阈值 `600 ms` 保持同量级，说明触发判据在当前样本下既未出现系统性提前触发，也未出现明显迟滞触发。这一现象与 `CER = 0` 的结果相互印证，表明触发机制参数处于可用区间。

## 5.5 一致性验证与后端对比

事件级一致性验证结果显示，Python 与 C++ 后端均输出 `5` 个事件，事件不一致数为 `0`。数值级一致性验证中，标定与平滑相关比较均未出现不一致记录，整体 `mismatch_count = 0`。以标定拟合指标为例，`mae_x = 0.002606`，`mae_y = 0.002418`，`rmse = 0.002783`，`max_abs_error = 0.004844`，说明关键数值口径在两后端间保持一致。

在运行时间方面，对同一示例数据重复 `5` 次测试后，Python 平均耗时 `152.49 ms`，C++ 平均耗时 `159.68 ms`，均值差 `-7.19 ms`。该结果表明当前小样本条件下两者时间差异有限，且不存在单向明显优势。结合一致性结果可以认为，本阶段后端切换的主要价值在于保持输出语义一致和支持架构替换，而非追求单次短序列测试下的极限速度。

从工程验证角度看，本节重点不在“谁更快”，而在“同一输入是否得到同一行为语义”。在可替换架构中，只有先保证事件与最终状态一致，后续性能优化才具有比较基础。当前结果已经满足这一前提，因此后续性能工作应转向长时序、多任务条件下的瓶颈定位，而不是继续在短序列样例上放大毫秒级差异。

## 5.6 失败案例与有效性讨论

为分析系统边界，本文对同日两次 Python 会话进行对比。成功会话中发生候选选择并完成提交，最终 `CER = 0`；失败会话中虽完成字符输入，但未执行提交动作，文本停留在组合区，按提交文本口径计算得到 `CER = 1.0`。该结果说明当前评估体系对“提交动作完成度”高度敏感，系统误差并非全部来自坐标质量，还与交互流程设计直接相关。

该案例提供两点启示：一是交互层应强化提交反馈机制，降低“输入完成但未提交”的失败概率；二是评估层可在主指标外增加辅助口径（如含组合区文本），以区分“识别失败”与“交互未闭环”两类问题。

该案例还揭示了论文实验中常见的口径问题：若评价指标仅统计最终提交文本，部分“近完成状态”会被归为完全失败，导致系统能力被低估；若只统计组合区文本，又会掩盖交互闭环缺陷。因此，本研究采用主指标与辅助指标并行的策略更为合理，即以提交文本口径作为主结论，以组合区口径作为诊断参考。

从外部有效性角度看，本章仍存在样本规模有限、任务文本较短、被试数量不足等限制。因此，结论更适用于当前实验条件下的工程可行性论证，后续仍需通过多被试、多轮次和长时任务实验进一步增强统计稳健性。

## 5.7 本章小结

本章通过基础链路、闭环输入、一致性对比和失败案例四类实验，对本文方法与系统进行了量化验证。结果表明，所提优化链路能够在实时条件下支撑眼控输入任务，且跨后端输出保持一致。与此同时，案例分析也揭示了当前系统在提交交互和样本规模上的局限，为后续优化方向提供了明确依据。
